## Context

当前 `SubscriptionCenterScreen.tsx` 已经具备较完整的认购中心功能，包括方案拉取、Tab/滑动切换、订单列表、确认认购、钱包状态和反馈提示；但它的视觉体系仍停留在旧版深色金属风，与已完成升级的首页明显不一致。  
由于首页已经建立了新的浅色金白视觉语言，这次认购中心更适合在既有交互骨架上做样式与信息呈现重构，而不是重新设计流程或拆掉现有状态管理。

## Goals / Non-Goals

**Goals:**
- 让认购中心在视觉上与新版首页形成统一的品牌感和精致度
- 保留现有认购、订单、滑动切换、钱包和反馈交互
- 提升方案信息、权益字段和订单信息的可读性与层次
- 尽量复用首页已建立的浅色卡片、金色强调和黑金 CTA 模式

**Non-Goals:**
- 不改动认购 API、订单 API 或支付/下单协议
- 不重构方案切换、滑动逻辑和核心状态机
- 不新增依赖或引入新的设计系统框架
- 不在本次变更中扩展新业务模块或新增认购功能

## Decisions

### 1. 复用现有交互结构，只升级视觉表达
- 决策：继续沿用 `SubscriptionCenterScreen` 现有的数据请求、方案切换、拖拽滑动、订单和确认认购逻辑，只对容器层、卡片层和文案层级做视觉重构。
- 原因：现有认购流程已经成型，重构业务交互会放大回归成本。
- 备选方案：在视觉升级同时重做布局和状态管理。该方案收益不高，风险过大。

### 2. 认购中心采用与首页一致的浅色金白主视觉
- 决策：将当前深色背景替换为浅色底色、白色主卡、金色高亮、黑金 CTA 和更轻的阴影体系，统一与首页风格。
- 原因：这是提升整体产品一致性的关键，也是用户最直观能感知到的变化。
- 备选方案：保留深色认购中心，只局部微调。这样仍会与首页形成割裂。

### 3. 强化“方案选择 + 主卡片 + 订单信息”的三层信息架构
- 决策：把方案切换区、主方案卡片、订单/记录区分别做出更清晰的视觉分层，并通过高亮态、徽标、字段对比和留白改善可读性。
- 原因：认购中心信息密度高，仅换背景不足以解决浏览压力。
- 备选方案：只修改配色不改信息层级。这样页面依然容易显得拥挤。

### 4. 与首页共享可复用视觉模式
- 决策：对齐首页已经确定的顶部导航、白卡片、浅金提示块、黑底按钮和细分隔线风格，必要时在共享组件或样式模式上复用。
- 原因：减少页面之间的重复设计判断，也让后续页面升级更可持续。
- 备选方案：认购中心单独定义一套视觉规则。这样维护成本更高。

## Risks / Trade-offs

- [视觉升级后内容密度仍偏高] → 通过字段分组、强调主指标和控制卡片层次来缓解
- [现有页面过长且状态较多] → 优先在原文件中做局部重构，必要时再提炼局部子组件
- [首页与认购中心共享头部但场景不同] → 共用基础视觉语言，同时允许认购中心保留少量场景化样式
- [已购态/登录态样式容易遗漏] → 实现阶段逐一验证未登录、已登录、已购、无订单等状态

## Migration Plan

这是纯前端展示层升级，不涉及接口迁移。上线方式为直接替换现有认购中心页面实现；如出现问题，可回滚本次前端改动恢复旧版样式。
