## Context

当前 `CommunityScreen.tsx` 已经具备完整的社区页基础能力，包括用户信息拉取、直推列表拉取、邀请码展示、邀请链接复制和顶部导航联动；但页面视觉仍停留在旧版深色风格，与已完成升级的首页和认购中心不一致。  
由于前两个页面已经建立了新的浅色金白视觉语言，这次社区页更适合在保留现有结构与状态逻辑的基础上做展示层重构，而不是重写数据流或改造交互模型。

## Goals / Non-Goals

**Goals:**
- 让社区页与新版首页、认购中心形成统一的品牌感与高级感
- 保留现有邀请链接、复制反馈、团队统计和直推列表等业务行为
- 提升邀请卡、数据卡和列表区的视觉层次、可读性与精致度
- 尽量复用前两个页面已验证过的浅色卡片、金色强调、深色 CTA 和留白节奏

**Non-Goals:**
- 不改动 `getUserInfo`、`getZhiList` 等 API 调用协议
- 不重构社区页核心状态管理与业务回调
- 不新增依赖或引入新的页面级框架
- 不在本次变更中扩展新的社区业务功能

## Decisions

### 1. 保留当前页面骨架，只重构展示层
- 决策：继续沿用 `CommunityScreen` 现有的数据请求、链接生成、复制反馈和列表渲染逻辑，只升级背景、容器、卡片和文案层级样式。
- 原因：社区页业务逻辑简单清晰，当前需求重点是统一视觉，而不是调整流程。
- 备选方案：同时重构数据流或拆分多个新组件。该方案收益有限，且会增加回归风险。

### 2. 社区页统一采用浅色金白主视觉
- 决策：把当前深色背景、深色半透明卡片替换为浅色底、白色主卡、柔和金色强调和轻量阴影体系，与首页和认购中心保持一致。
- 原因：社区页属于主流程页面之一，视觉割裂会直接影响整体产品一致性。
- 备选方案：仅局部改按钮和标题。这样无法解决页面整体气质不统一的问题。

### 3. 强化“三段式信息结构”
- 决策：将页面拆解为页头说明、邀请卡与团队数据、直推列表三个主要层次，通过容器分组、标签高亮、数据优先级和留白节奏提升浏览效率。
- 原因：社区页信息量虽然不大，但存在功能块与列表混排，旧样式下识别效率一般。
- 备选方案：保持当前单层铺排，只换颜色。这样可读性提升有限。

### 4. 列表区域沿用统一卡片语言但保留数据表格可读性
- 决策：在直推列表中保留“地址 / 团队 / 注册时间”的数据表头和行式结构，但以浅色卡片、边框、序号徽标和更清晰的文字对比来重做样式。
- 原因：这块区域本质上是数据浏览，不能因为追求视觉效果而牺牲表格信息清晰度。
- 备选方案：把列表改成复杂信息卡。这样会降低扫描效率。

## Risks / Trade-offs

- [浅色化后列表信息对比不足] → 通过更明确的主次文字颜色、细边框和数字对齐来保证可读性
- [复制邀请链接区域过于装饰化] → 优先保留链接展示与 CTA 明确性，再补充视觉质感
- [共享顶部导航在不同页面存在语境差异] → 继续复用统一头部，只在社区页内容区做场景化样式调整
- [加载态、空态和未登录态容易被忽略] → 实现阶段逐一验证 loading、空列表和无邀请码等状态显示

## Migration Plan

这是纯前端展示层升级，不涉及接口迁移。上线方式为直接替换当前社区页实现；如出现问题，可回滚本次前端改动恢复旧版社区页样式。
