## Context

当前首页实现在 `src/features/figma/pages/HomeScreen.tsx`，整体采用深色背景、金色描边和玻璃态卡片；而最新 Figma 首页已经切换为浅色金白视觉，页面背景、区块卡片、主按钮、合作伙伴区和联系方式区都明显更新。  
这次改版的特殊点在于：首页的信息架构、区块顺序和交互入口并未推倒重来，主要变化集中在视觉风格、资源素材和区块内容承载形式。因此适合在现有页面骨架上做 Figma 对齐，而不是重新设计一个新的页面容器或状态模型。

## Goals / Non-Goals

**Goals:**
- 让首页在移动端视觉上尽量贴近 Figma 节点 `134:744`
- 保留现有 `HomeScreen` 的 props 契约、section refs 和外部事件回调
- 以项目现有的 React + Tailwind 4 方案完成视觉重构，不新增样式体系
- 为首页资源更新提供清晰接入点，便于替换 Hero、合作伙伴和其他新版素材

**Non-Goals:**
- 不改动首页对外暴露的业务交互行为
- 不新增接口、路由或全局状态逻辑
- 不直接照搬 Figma 生成的绝对定位代码到项目中
- 不在本次变更中扩展桌面端专用布局或重新定义整个文案国际化方案

## Decisions

### 1. 保留现有页面结构与交互接口，只重构视觉层
- 决策：继续使用现有 `HomeScreen` 组件作为首页承载入口，保留 `TopNavigation`、`NoticeBar`、`NoticeDetailSheet`、钱包连接、语言切换、认购入口和各 section refs。
- 原因：这次改版以视觉和资源更新为主，复用现有交互契约可以降低联动回归风险。
- 备选方案：新建一套首页容器并重新接线所有交互。该方案改动面更大，不适合当前“布局不变、UI 更新”的需求。

### 2. 用 Tailwind 语义化组合重建新版卡片体系，而不是照抄 Figma 绝对定位
- 决策：以当前页面的顺序布局为基础，使用 Tailwind 的间距、圆角、阴影和背景色组合来还原新版视觉，必要时抽出局部展示组件或复用块级样式模式。
- 原因：Figma 导出的参考代码偏绝对定位，不适合维护；而现有页面已经是顺序流式布局，能更稳地适配内容和移动端高度变化。
- 备选方案：按照 Figma 导出的绝对尺寸逐像素拼装。该方式维护成本高，且对文案换行和未来内容调整不友好。

### 3. 新资源以本地可控资产为准，Figma 只作为视觉参考
- 决策：Hero 图、合作伙伴 Logo、图标与插图优先落到项目本地资源或现有数据层，再由页面消费，不直接依赖 Figma MCP 返回的临时远程地址。
- 原因：Figma 资产链接具有时效性，直接线上依赖会带来失效风险。
- 备选方案：直接使用 Figma 远程资源地址。该方案不稳定，且不利于后续压缩和版本管理。

### 4. 内容更新与样式重构一起推进，但限制在首页展示层
- 决策：将新版设计中已经调整的资源、配送/生态说明、合作伙伴和联系方式展示一并纳入首页改版；但数据结构只做最小调整，以支持新的展示形式。
- 原因：如果只换皮不换内容，最终页面仍会与新版设计不一致。
- 备选方案：先做纯样式改版，后续再换资源内容。该方案会导致两次首页返工。

## Risks / Trade-offs

- [Figma 与现有文案来源不完全一致] → 优先以新版设计展示效果为目标，必要时只调整首页展示层的数据映射，不扩大到全局文案系统
- [单文件页面继续膨胀] → 在实现时优先提炼重复的区块样式或局部卡片组件，避免 `HomeScreen.tsx` 可读性继续恶化
- [移动端相对布局与 Figma 像素值存在偏差] → 用截图对照做视觉校正，重点检查标题换行、卡片高度和区块间距
- [资源替换不完整导致视觉新旧混用] → 在实现阶段建立资源清单，逐一核对 Hero、合作伙伴、联系方式和区块插图

## Migration Plan

这是一次前端展示层改版，不涉及数据库或接口迁移。上线策略为直接替换当前首页实现；如果出现问题，可通过回滚本次前端变更恢复旧版首页。
